PRODUCT OVERVIEW
----------------

DriveScore is an AI-powered NEET mock-test analysis and gamified learning
platform designed for coaching centres, schools, and individual aspirants
preparing for the Indian NEET medical entrance exam.

The core differentiator is the Diagnosis Engine — a deterministic algorithm
that analyzes per-question timing and answer patterns to explain WHY marks
were lost, not just report a score.

Built as a Next.js PWA (Progressive Web App) with Supabase backend.
Also available as an Android app via Trusted Web Activity (TWA).

Product intent: "Don't just score it. Diagnose it."

1. AUTHENTICATION & ROLE SYSTEM
==============================================================================

Three distinct user roles with row-level security (RLS) enforced at the
database level.

STUDENT ROLE
  - Individual test-takers preparing for NEET.
  - Can take assigned mocks, practice lesson-by-lesson, view personal reports.

TEACHER ROLE

  - Full CRUD on questions + mocks within their centre.
  - Can publish/unpublish assignments to batches.
  - View class dashboard: per-student ratings, leaderboard, diagnosis breakdown,
    weak chapters.
  - Key files: app/teacher/page.tsx, lib/db/teacher.ts

ADMIN ROLE
  - Platform super-admin (DriveScore company). No centre affiliation.
  - Cross-centre operations: create centres, manage teachers, run analytics.
  - Manage global question bank accessible to all students for practice.
  - Key files: app/admin/page.tsx, lib/db/adminAnalytics.ts

AUTHENTICATION DETAILS
  - Supabase Auth (Postgres-backed user management).
  - Login via email + password; password reset via email link (expires 1 hour).
  - Auth middleware redirects unauthenticated users to /login.
  - Service-key (server-side) for privileged operations (grading, rating
    updates, user creation) — never exposed to the browser.
  - User-scoped client for RLS-enforced access.

2. THE DIAGNOSIS ENGINE  (Core Differentiator)
==============================================================================

A pure, deterministic algorithm classifying each wrong answer into one of
seven categories. No AI/LLM; fully unit-tested.

File: lib/diagnose.ts

CLASSIFICATION LOGIC
  1. Unattempted                         → TIME_MANAGEMENT
  2. Correct + slow (> 140% of par)      → TOO_SLOW
  3. Correct + normal pace               → SOLID  (no problem)
  4. Wrong + Easy difficulty             → CARELESS  (a slip)
  5. Wrong + rushed (< 40% of par)       → GUESS  (answered too fast)
  6. Wrong + Medium/Hard at normal pace  → CONCEPT_GAP  (genuine knowledge gap)
  7. First answer correct, final wrong   → SELF_DOUBT  (changed a right answer)

SELF_DOUBT DETAIL
  - Triggered when student changes a CORRECT first answer to a WRONG final one.
  - Requires first_answer_index tracking (migration 0015).
  - Confidence: 95% (high certainty — instinct was provably right).

CONFIDENCE SCORING
  - Each diagnosis also gets a 0–100 confidence score.
  - Uses only within-attempt signals: timing vs. par, difficulty,
    first-vs-final answer.
  - No cross-attempt or cross-student data (privacy-first design).

PER-CATEGORY ADVICE SHOWN TO STUDENTS
  CONCEPT_GAP    "Re-learn from NCERT before practising more"
  GUESS          "Slow down; use elimination before committing"
  SELF_DOUBT     "Trust your first instinct; only change for a concrete reason"
  CARELESS       "Slow down on easy questions"
  TOO_SLOW       "Practise these for speed"
  TIME_MANAGEMENT "Work on pacing across the full paper"
  SOLID          "Correct and on time — keep it up"

SPEED ARCHETYPES  (whole-attempt pacing personality)
  SNIPER    Accurate but deliberate. Advice: build speed.
  GAMBLER   Fast but loose. Advice: slow down and eliminate options.
  PANICKER  Strong start, accuracy drops sharply in final fifth.
            Advice: practise pacing the full paper.
  BALANCED  Speed and accuracy well matched. Advice: keep building both.
  Requires ≥5 attempted questions to assign an archetype.


==============================================================================
3. RATING SYSTEM  (Skill Progression)
==============================================================================

Fair Elo-based rating reflecting the hardest difficulty tier a student
reliably beats. Points farming is mathematically impossible.

File: lib/rating.ts

ELO MECHANICS
  - Every student starts at rating 1000.
  - Each question is an "opponent" with a fixed rating by difficulty:
      Easy   → 800
      Medium → 1100
      Hard   → 1400
  - Correct vs. question above your rating → large gain.
  - Correct vs. question far below your rating → ~zero gain (farming blocked).
  - Wrong vs. easy question → meaningful loss.
  - Wrong vs. hard question → tiny loss.

K-FACTOR (Volatility)
  - First 30 rated questions per subject (calibration): K = 32  (fast move).
  - After calibration: K = 16  (stable, slow movement).

LEVELS  (Sticky — Road to the White Coat)
  Aspirant    (0+)
  Achiever    (900+)
  Scholar     (1050+)
  Ranker      (1200+)
  Topper      (1350+)
  White Coat  (1500+)

  Sticky rule: a level is only LOST once rating falls 25 points below its
  floor (demotion buffer prevents flickering at boundaries).

OVERALL RATING  (NEET-weighted)
  Mirrors NEET mark split:
    Biology 50%  |  Physics 25%  |  Chemistry 25%
  Single headline "Overall" rating answers "How NEET-ready am I?".

PER-CHAPTER RATINGS
  - Subject × chapter granularity (e.g. "Physics / Ray Optics").
  - Same Elo engine; drives chapter-by-chapter weakness discovery.
  - Used by practice page to show strongest and weakest lessons.
  - Migration: 0012

ANTI-FARMING
  - Once a question is answered correctly, subsequent correct answers on the
    SAME question yield delta=0.
  - Checked against all prior attempts for that student × question.
  - Idempotency: rating_events table has unique(attempt_id, question_id).
    Re-submitting an attempt never double-applies ratings.


==============================================================================
4. MASTERY ROAD  (Progressive Gate System)
==============================================================================

Chapter-by-chapter progression map with four difficulty gates per chapter.
NOT a "% to NEET" predictor — purely a progression visualizer.

File: lib/mastery.ts

GATE STRUCTURE  (per chapter)
  Foundation (Easy) → Application (Medium) → NEET-Level (Hard) → Mastery

  Gates unlock in order; previous gate must be cleared before the next opens.
  Hard questions feed BOTH NEET-Level and Mastery gates simultaneously.

GATE CLEARING  (Earned, Not Lucky)
  - Requires: 4 STRONG correct answers + 70% accuracy at that tier +
    minimum 2 separate sessions.
  - One lucky answer is NOT enough; consistency required.

GATE REGRESSION  (Recent Decay)
  - A cleared gate moves back to NEEDS_REINFORCEMENT if recent accuracy
    drops below 55%.
  - Checks last 6 answers at that tier; minimum 3 recent answers before
    regression can trigger.

FRONTIER PRESCRIPTION  (Next-Step Quest)
  - Server finds the single next frontier gate (earliest locked / regressed).
  - Recommends 8 targeted questions at that gate's difficulty.
  - Student taps "Start 8 Questions" → personal practice mock generated.
  - After completion: same report + diagnosis + rating pipeline runs.

REVISIT PROMPTS
  - Regressed gates flagged "Worth revisiting — recently slipped".
  - Student can revisit any gate, not just the frontier.

5. PRACTICE PATHWAYS
==============================================================================

Two self-directed student practice flows reusing the existing
test → attempt → report pipeline. Both generate personal mocks on the fly.

FULL NEET MOCK  (Practice / Bank)
  - 180-question shuffled paper (45 Physics + 45 Chemistry + 90 Biology).
  - Sampled from the global question pool (centre_id IS NULL).
  - Language-filtered: Tamil students only see questions with body_ta IS NOT NULL.
  - Each "start" generates a fresh mock (kind='bank', max_attempts=1).
  - Same TestRunner component as assigned mocks.

LESSON PRACTICE  (Practice by Lesson / Climb)
  - Pick subject + chapter → focused 8-question test on that topic.
  - Generates at the frontier's difficulty tier.
  - Same report + diagnosis + rating flow.

ADAPTIVE CLIMB  (Gamified Lesson Practice)
  - Infinite adaptive difficulty; questions stream one at a time until HP = 0.
  - Correct answer → difficulty rises, HP healed (+8 HP), XP gained, streak +1.
  - Wrong answer  → difficulty drops, HP reduced (Easy –25, Medium –15, Hard –10).
  - Run ends at HP = 0 or when chapter pool exhausted.
  - All answered questions persisted as a real attempt → full report/diagnosis.
  - XP formula: base (10 × difficulty_multiplier) × speed_bonus × streak_multiplier.

PRACTICE PAGE  (lib/db/practice.ts)
  - Dynamically lists global syllabus (subjects → chapters → question counts).
  - Shows per-chapter rating badge (e.g. "Scholar" next to each chapter).
  - Two tracks: Past Papers (pyq) and AI Practice (ai).

6. ATTEMPT SUBMISSION & GRADING  (Server-Side)
==============================================================================

All grading happens server-side with the service key.
Answer keys NEVER reach the browser.

File: app/actions.ts → submitAttempt()

FLOW
  1. Student finishes mock (all answered or time skipped).
  2. Client sends answers; server re-fetches questions using service key.
  3. Server checks access: is mock published for student's batch?
  4. Server enforces attempt limit: has student exceeded max_attempts?
  5. Server grades using buildReport() (pure function).
     Output: score, per-question diagnosis, 9 report sections.
  6. Server persists attempt + answers to DB.
  7. Server updates student ratings (best-effort; failure logs but
     doesn't block the submission).
  8. Client redirected to /report?attempt=<id>.

MAX ATTEMPTS
  - Per-mock configurable (default 1; teachers can set higher).
  - Teacher can manually "unlock retake" for any student.

PER-QUESTION TIMING CAPTURE
  - Client records shownAt timestamp when each question appears.
  - On submit: elapsed = now − shownAt (seconds).
  - First picked index also recorded (for SELF_DOUBT detection).
  - All client-side; transmitted to server with each answer.

14. TAMIL TRANSLATION PIPELINE
==============================================================================

Chat-AI workflow to translate 3,996 English NEET questions to Tamil.
Zero Anthropic API cost — uses Claude Code chat tokens instead.

Files: scripts/pipeline-chat-fetch.ts, pipeline-chat-store.ts,
       scripts/patch-queue.ts, scripts/show-queue.ts

PIPELINE STEPS  (per batch of 50)
  1. npm run translate:fetch -- --subject Physics --status none --limit 50
     Writes .translation-queue.json with translation: null for each question.
  2. Claude (in chat) reads all questions, produces Tamil translations.
  3. Write scripts/patch-queue.ts with the translations map.
  4. npx tsx scripts/patch-queue.ts
     Injects translations into the queue file.
  5. npm run translate:store
     Validates + stores to DB + clears queue file.

DB TABLES
  - tamil_knowledge_chunks  : pgvector RAG store (textbook passages).
  - tamil_glossary          : forced term map (English → Tamil, per-subject).

QUESTION COLUMNS (added by migration 0017, renamed by 0019)
  - body_ta        (renamed from tamil_question_text)
  - options_ta     (renamed from tamil_options)
  - explanation_ta (renamed from tamil_explanation)
  - tamil_status   : none | ai_drafted | review_required | approved | rejected
  - tamil_confidence_notes   : model self-reported metadata
  - tamil_validation_result  : objective validator output
  - tamil_error_log          : parse/API/validator failures

OBJECTIVE VALIDATOR  (pipeline-chat-store.ts)
  Weights: risk_words_preserved=30, glossary_match_rate=25,
           option_count_match=20, number_count_match=10,
           unit_preserved=10, formula=5, math=5.
  Score < 80 → auto-set to review_required.

KNOWN VALIDATOR FALSE POSITIVES
  1. unit_preserved fails on almost every question.
     Root cause: Ω encoding mismatch (U+2126 vs U+03A9) + regex picks up
     English words after numbers ("10 If", "3 and") as unit candidates.
     Impact: -10 points on every question; score 95 instead of 100.
  2. risk_words_preserved false positives on "MAXIMUM", "MINIMUM" in
     descriptive context.
     Impact: some correct translations scored 65 → review_required.
  Actual translations are correct despite lower scores.

TEXTBOOK CHUNKS LOADED
  - Physics Vol 1 (Class 11, 2024 edition): 999 passages.
  - Chapters: Units & Measurements, Kinematics, Laws of Motion,
    Work/Energy/Power, Rotational Motion.
  - No embeddings — retrieval uses chapter column filter directly.

PROGRESS (as of 2026-06-18)
  Physics    : 98 translated  (~902 remaining)
  Chemistry  :  0 translated  (~1000 remaining)
  Biology    :  0 translated  (~2000 remaining)
  Total DB   : 3,996 questions

LANGUAGE LOCK
  - Each student picks English ('en') or Tamil ('ta') at signup.
  - Stored in profiles.preferred_language.
  - DB trigger (lock_preferred_language, migration 0019) blocks ANY change
    once set — immutable.
  - Tamil students: only see practice questions where body_ta IS NOT NULL.
  - Both _en and _ta columns coexist in the same questions table.
    Query layer fetches the right version based on student's preference.

15. GAMIFICATION
==============================================================================

XP + Streaks + Levels + Mastery Road form a progression loop
without being "just points".

LEVEL SYSTEM  (home screen Level Card)
  - Six visible levels: Aspirant → Achiever → Scholar → Ranker →
    Topper → White Coat.
  - Sticky: once earned, lost only if rating falls 25 pts below the floor.
  - Visual roadmap: horizontal track with 6 animated nodes.
    Completed = filled ✓, Current = pulsing YOU pin, Future = dimmed.

XP  (Climb mode only)
  Formula: base × difficulty_multiplier × speed_bonus × streak_multiplier
    base                = 10 XP
    difficulty_multiplier : Easy=1, Medium=1.5, Hard=2
    speed_bonus         : 1.5× if answered within par time
    streak_multiplier   : +10% per consecutive correct (capped at 5)
  Total XP displayed in post-game report.

STREAKS  (Climb mode only)
  - Current streak: consecutive correct answers.
  - Resets to 0 on any wrong answer.
  - Best streak for the session shown in report.

HP  (Climb mode only)
  - Start: 100 HP.
  - Healing: +8 HP per correct answer.
  - Damage: Easy −25, Medium −15, Hard −10.
  - Run ends when HP = 0 or question pool exhausted.

MASTERY ROAD PROGRESSION
  - Four gates per chapter (Foundation → Application → NEET-Level → Mastery).
  - Frontier quest: the one next gate to clear.
  - Regressed gates: "Worth revisiting — recently slipped".
  - NOT a "% to doctor" or "% to NEET" meter (deliberately avoided).

20. SECURITY & PRIVACY
==============================================================================

ANSWER KEY PROTECTION
  - answer_index NEVER sent to browser (excluded from PublicQuestion type).
  - questions table: NO select policy for user-scoped clients (deny-all).
  - Questions fetched server-side with service key only.
  - answer_index stripped before serialization.

DATA ISOLATION
  - Row-level security (RLS) at Postgres level on ALL user-facing tables.
  - Each row tagged: student_id / centre_id / batch_id.
  - Policies enforce visibility; service key bypasses for trusted ops.

PRIVACY BY DESIGN
  - Confidence scoring uses only within-attempt signals — no cross-student
    comparison data.
  - Tamil translation uses only textbook + exam data (no student data).
  - One student CANNOT see another student's report (RLS enforced).

SESSION MANAGEMENT
  - Supabase SSR client handles server-side sessions.
  - Auth cookies: secure, httpOnly, SameSite.
  - Token refresh automatic.

23. NOTABLE IMPLEMENTATION DETAILS
==============================================================================

ATTEMPT IDEMPOTENCY
  - rating_events: unique(attempt_id, question_id).
  - Re-submitting an attempt never double-applies ratings.
  - Client uses POST-Redirect-GET on question picker to avoid double-submit.

MASTERY ROAD CHAPTER ORDERING
  - Chapters sorted by NCERT order (not alphabetical).
  - chapterRank() in lib/questions/chapters.ts returns ordinal per
    subject + chapter.
  - Enables consistent, intuitive progression UI.

PAGINATION FOR LARGE QUESTION POOLS
  - Supabase caps single select() at 1000 rows.
  - listGlobalSyllabus: loops from=0; from+=1000 until data.length < 1000.
  - Supports accurate counts even for 10k+ global pool.

RANDOM SAMPLING
  - shuffle<T>(): Fisher-Yates in-place shuffle; returns new array.
  - Used for full mock + lesson practice question selection.

ARCHETYPE DETECTION  (PANICKER check)
  - Splits attempt into 5 equal quintiles.
  - Computes per-quintile accuracy.
  - PANICKER: final quintile accuracy drops ≥ 20 percentage points vs. first.

SMOOTH ANIMATIONS
  - Framer Motion spring animations for progress, level roadmap,
    confetti, card reveals.
  - Parallax effect on landing page (CinematicBackground / Three.js scene).
  - useReducedMotion() respected everywhere: all animations have a
    reduce-? branch that skips animation.

SUMMARY
==============================================================================

DriveScore is a diagnostic learning platform built around a deterministic,
explainable algorithm that transforms NEET mock scores into actionable
learning paths.

It combines:
  - Skill-rating (Elo)              → fair, transparent progression
  - Diagnosis Engine                → WHY marks were lost, not just HOW MANY
  - Mastery Road                    → chapter-by-chapter gate progression
  - Gamification (XP/streaks/HP)    → engagement in Climb mode
  - Bilingual pipeline (EN + TA)    → Tamil medium students included
  - Multi-tenancy (centres/batches) → B2B coaching centre product
  - Three report views              → student / teacher / parent

Core product insight: "Diagnose, not just score."
Core technical insight: server-side grading + pure deterministic logic +
row-level security = safe, explainable, privacy-respecting.
